# Step 15 — Get Buy-In Before Coding ⭐

## 🎯 Purpose
After explaining the plan, EXPLICITLY ask if it's good before typing.
This is the single most overlooked step.

## 🧠 What interviewer is testing
Whether you collaborate or just barrel forward.
Also: gives them the chance to course-correct you BEFORE you waste 20 minutes.

## 🇬🇧 English phrases (memorize these EXACTLY)

```
"Does this approach sound good before I start coding?"
"Are you happy with this approach? Should I go ahead and implement it?"
"Before I code — does this direction look right to you?"
"Any concerns with this approach before I start writing it?"
```

## 📚 Required knowledge

**This is a 5-second step that can save 20 minutes of wasted code.**

What can happen at buy-in:
1. ✅ "Yes, go ahead." → Code with confidence.
2. 🟡 "It works, but can you do it without extra space?" → Adjust before coding.
3. 🔴 "Actually I'd like to see a different approach." → Saves you from coding the wrong thing.

**Without buy-in**: you might code 20 minutes, then they say "I had something else in mind" → you're cooked.

## 🔑 What if they push back?

Use Step 16's transition:
> *"That's a great point. Let me think about how to do it without extra space."*

NEVER:
- Defend your approach against their hint
- Get flustered
- Try to rush through the code

## ⚠️ Common mistakes

- Forgetting to ask → coding wrong direction
- Asking but not WAITING for an answer
- Asking weakly: "uhh ok I'll start now?" → no clear pause for them to respond

## 🔁 Drill

Practice saying each phrase out loud, with confident pause after:

1. *"Does this approach sound good before I start coding?"* — pause, count to 3.
2. *"Are you happy with this direction?"* — pause, count to 3.
3. *"Any concerns before I implement?"* — pause, count to 3.

The pause is the point. Don't fill silence — let them respond.
