## 🟦 CLARIFY (4 min)

### 1. Restate

> **You:** *"So I need to design a cache class with a fixed capacity.
> When a key is accessed or written, it becomes the most recent.
> When the cache is full and a new key is added, the LEAST recently used key gets evicted.
> Both get and put should be O of one. Right?"*
> **Interviewer:** *"Correct."*

### 2. Methods — what each one does

> **You:** *"Two questions about the methods.
> First, get(key): if the key exists, return its value AND mark it as recently used. If not, return what?
> Second, put(key, value): if the key exists, update the value. If not, insert. When full, evict LRU. Right?"*
> **Interviewer:** *"get returns -1 if not found. Yes for put."*

### 3. Capacity

> **You:** *"What's the capacity range? And is it fixed at construction?"*
> **Interviewer:** *"Capacity is between 1 and 3000. Fixed at construction."*

### 4. Edge cases — clarify

> **You:** *"A few edge cases — what if I put with a key that already exists?
> Does updating an existing key count as a 'use'?"*
> **Interviewer:** *"Yes. Both get and put should mark the key as recently used."*

### 5. Constraints

> **You:** *"Thread safety needed?"*
> **Interviewer:** *"No, single-threaded is fine."*

### 6. Example

> **You:** *"Could I get an example?"*
>
> **Interviewer:**
> *"Capacity 2.*
> *put(1, 100). put(2, 200).*
> *get(1) returns 100. Now key 1 is most recent, key 2 is least.*
> *put(3, 300) evicts key 2.*
> *get(2) returns -1."*
>
> **You:** *"Got it. Order matters: [LRU=2, 1] → after get(1) → [2, 1=MRU] → put(3) evicts 2 → [1, 3=MRU]."*

## 🟦 PLAN — the DEEP part of design problems (5 min)

### 7. Naive approach — and why it fails

> **You:** *"Naive: just use a dict. get and put are O of one for value lookup,
> BUT a dict doesn't track ORDER of access. To find the LRU when full, I'd scan all entries — O of n. So that fails the O of one requirement."*

### 8. What we need — break it down ⭐

> **You:** *"Let me list what I actually need:*
> *1. Fast lookup by key — that's a hash map.*
> *2. Track which key is least recently used — that needs ORDER.*
> *3. When a key is touched, move it to the 'most recent' end — fast.*
> *4. When evicting, remove the 'least recent' end — fast.*
>
> *No single data structure gives all four. I need to combine two."*

### 9. The key insight ⭐⭐⭐

> **You:** *"Combine HASH MAP + DOUBLY LINKED LIST.*
>
> *The doubly linked list maintains order: head = most recent, tail = least recent.*
> *The hash map points key → node in the list.*
>
> *Why doubly linked? Because when I access a key, I get the node from the map in O of one,
> but to MOVE it I need to unlink it from the middle — that requires the previous pointer too.*
> *Singly linked won't work for O of one removal from the middle."*

### 10. Visual model

```text
HEAD <-> [key1, val1] <-> [key2, val2] <-> [key3, val3] <-> TAIL
         (most recent)                     (least recent)

cache (hash map):
    key1 -> Node(key1, val1)
    key2 -> Node(key2, val2)
    key3 -> Node(key3, val3)

On get(key2):
    1. Look up key2 in cache → get Node    O(1)
    2. Unlink Node from current position    O(1)
    3. Insert Node right after HEAD          O(1)

On put when FULL:
    1. lru = TAIL.prev (the actual last node)
    2. Unlink lru from list
    3. Delete cache[lru.key]
    4. Add new node to front
```

> **You:** *"I also use sentinel HEAD and TAIL nodes.
> They aren't real entries — they exist to avoid null checks at the boundary.
> The first real node is `HEAD.next`. The last real node is `TAIL.prev`."*

### 11. Complexity upfront

> **You:** *"Both get and put are O of one — amortized.
> Hash map operations are O of one.
> Linked list insert/remove given a node reference is O of one.
> Space is O of capacity."*

### 12. Buy-in

> **You:** *"Does this approach sound good before I start coding?"*
> **Interviewer:** *"Go ahead."*

### 13.2 The code

```python
class _Node:
    """Doubly-linked list node."""
    __slots__ = ("key", "value", "prev", "next")

    def __init__(self, key: int, value: int) -> None:
        self.key = key
        self.value = value
        self.prev: "_Node | None" = None
        self.next: "_Node | None" = None


class LRUCache:
    def __init__(self, capacity: int) -> None:
        if capacity <= 0:
            raise ValueError("Capacity must be positive")
        self.capacity = capacity
        self.cache: dict[int, _Node] = {}

        # Sentinel nodes — they avoid null checks at boundaries
        self.head = _Node(0, 0)        # most-recent end
        self.tail = _Node(0, 0)        # least-recent end
        self.head.next = self.tail
        self.tail.prev = self.head

    # ---- Public API ----

    def get(self, key: int) -> int:
        if key not in self.cache:
            return -1
        node = self.cache[key]
        self._move_to_front(node)
        return node.value

    def put(self, key: int, value: int) -> None:
        if key in self.cache:
            node = self.cache[key]
            node.value = value
            self._move_to_front(node)
            return

        if len(self.cache) >= self.capacity:
            self._evict_lru()

        node = _Node(key, value)
        self.cache[key] = node
        self._add_to_front(node)

    # ---- Private helpers ----

    def _add_to_front(self, node: _Node) -> None:
        # Insert right after head sentinel
        node.prev = self.head
        node.next = self.head.next
        self.head.next.prev = node
        self.head.next = node

    def _remove(self, node: _Node) -> None:
        # Unlink node from its neighbors
        node.prev.next = node.next
        node.next.prev = node.prev

    def _move_to_front(self, node: _Node) -> None:
        self._remove(node)
        self._add_to_front(node)

    def _evict_lru(self) -> None:
        # The actual last node is right before the tail sentinel
        lru = self.tail.prev
        self._remove(lru)
        del self.cache[lru.key]
```

### 13.3 What to say while typing

- *"_Node class — just key, value, prev, next. I use __slots__ for a small memory win."*
- *"LRUCache constructor — validate capacity, init empty dict, set up head and tail sentinels linked to each other."*
- *"get — if key missing return minus one, otherwise move node to front and return value."*
- *"put — if key exists, update value and move to front. If at capacity, evict. Then create new node and add to front."*
- *"Helper add_to_front — link node right after head sentinel."*
- *"Helper remove — bypass node by linking its neighbors to each other."*
- *"Helper move_to_front — just remove plus add_to_front."*
- *"Helper evict_lru — node before tail sentinel is the LRU. Remove and delete from map."*

### 13.4 Naming choices

- `_Node` — leading underscore = "internal class, don't use externally"
- `cache` — the hash map. `map` would shadow Python's built-in
- `_add_to_front`, `_remove` — leading underscore = "private helper"
- Sentinels named `head`, `tail` — standard
- `_evict_lru` — verb-noun, says exactly what it does

> 💡 **Production rule applied here**: split the doubly linked list operations into 4 helpers instead of one giant method. Each helper does ONE thing. R2 graders look for this.

```python
        node.prev = self.head
        node.next = self.head.next
        self.head.next.prev = node
        self.head.next = node

    def _remove(self, node: _Node) -> None:

    def _move_to_front(self, node: _Node) -> None:
        self._remove(node)

        self._remove(lru)
        del self.cache[lru.key]
```

**Complexity:**

- `get`: **O(1)** amortized
- `put`: **O(1)** amortized
- Space: **O(capacity)**

**Shortcut alternative — using OrderedDict** (mention this only if asked):

```python
from collections import OrderedDict


class LRUCache:
    def __init__(self, capacity: int) -> None:
        self.capacity = capacity
        self.cache: OrderedDict[int, int] = OrderedDict()

    def get(self, key: int) -> int:
        if key not in self.cache:
            return -1
        self.cache.move_to_end(key)        # mark as most recent
        return self.cache[key]

    def put(self, key: int, value: int) -> None:
        if key in self.cache:
            self.cache.move_to_end(key)
        elif len(self.cache) >= self.capacity:
            self.cache.popitem(last=False)  # evict LRU (front)
        self.cache[key] = value
```

> 💡 **In an interview, prefer the from-scratch version.**
> The OrderedDict version is "too easy" — interviewer wants to see your understanding.

## 🧠 Pattern: Hash Map + Doubly Linked List — what to remember

| When to use | Need O(1) lookup AND O(1) ordered removal |
|---|---|
| Why dict alone fails | No ordering |
| Why list alone fails | O(n) lookup |
| Why DOUBLY linked | Need prev pointer for O(1) removal from middle |
| Why sentinels | Avoid null checks at boundaries |
