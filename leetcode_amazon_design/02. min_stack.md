## 🟦 CLARIFY (3 min)

### 1. Restate

> **You:** *"So I need a stack with the usual operations — push, pop, top — plus a getMin that returns the smallest element currently in the stack.
> All four must be O of one. Right?"*
> **Interviewer:** *"Yes."*

### 2. Methods — what each one does

> **You:** *"A few clarifications:*
> *push(val) — adds val to the top.*
> *pop() — removes top, returns nothing or returns the popped value?*
> *top() — returns the top element without removing.*
> *getMin() — returns the minimum, also without removing.*
> *And what if pop or top is called on an empty stack?"*
>
> **Interviewer:** *"pop returns void. top returns the element. Operations are always called validly — assume the stack isn't empty."*

### 3. Input

> **You:** *"What types of values? Integers? Range?"*
> **Interviewer:** *"Integers. They can be negative or positive."*

### 4. Constraints

> **You:** *"Memory constraints?"*
> **Interviewer:** *"No, anything reasonable."*

### 5. Example

> **You:** *"Could I get an example?"*
>
> **Interviewer:**
> *"push(-2), push(0), push(-3).*
> *getMin() returns -3.*
> *pop().*
> *top() returns 0.*
> *getMin() returns -2."*
>
> **You:** *"OK so after push(-3), the min in the stack is -3. After we pop -3 off, the min reverts to -2. So we need to track historical mins too, not just current min."*

### 6. My own example

> **You:** *"Let me try push(5), push(3), push(3), push(7).*
> *getMin should return 3. Pop, getMin still 3 (one 3 left). Pop, getMin is 5. Right?"*
> **Interviewer:** *"Correct."*

## 🟦 PLAN (4 min)

### 7. Naive approach — and why it fails

> **You:** *"Naive approach 1: track a single 'current min' variable.
> But when I pop the current min, I lose information — I'd have to scan the rest of the stack to find the new min, which is O of n.*
>
> *Naive approach 2: scan the stack on every getMin.
> That's O of n per call. Both fail the O of one requirement."*

### 8. What we need ⭐

> **You:** *"For O of one getMin AFTER pops, I need to remember the min at each level of the stack.
> When I pop, the min should automatically revert to the previous min."*

### 9. The key insight ⭐⭐⭐

> **You:** *"Maintain a SECOND stack — the 'min stack' — that tracks the running minimum.*
>
> *On push: push to main stack, AND push min(new_val, current_min) onto min stack.*
> *On pop: pop both stacks together.*
> *On getMin: peek the top of min stack — that's always the current min."*

### 10. Visual model

```text
push(-2)         push(0)          push(-3)         pop()           pop()
─────            ─────            ─────            ─────           ─────
main: [-2]       main: [-2, 0]    main: [-2,0,-3]  main: [-2, 0]   main: [-2]
min:  [-2]       min:  [-2, -2]   min:  [-2,-2,-3] min:  [-2, -2]  min:  [-2]

getMin reads top of "min" stack: -3, then -2, then -2 — all O(1).
```

> **You:** *"Each entry in the min stack records the minimum AT THAT POINT in the main stack's history.
> When we pop, we automatically restore the previous min — no scanning needed."*

### 11. Complexity upfront

> **You:** *"All four operations are O of one. Space is O of n — we use one extra entry in the min stack per push. That's the trade-off: extra memory for constant-time getMin."*

### 12. Buy-in

> **You:** *"Does this approach sound good before I start coding?"*
> **Interviewer:** *"Go ahead."*

### 13.0 Quick check

> **You:** *"This is a design problem — I'll write a class. Just MinStack, no helper class needed for this one."*

### 13.1 Approach (recap)

> **You:** *"One class with two internal lists — `_stack` and `_min_stack`.
> Both grow and shrink together on push and pop."*

### 13.1.5 ⭐ Pseudocode-comments FIRST

```python

    def __init__(self):

        # 2. _min_stack — running minimums
        pass

    def push(self, val):

        # 3. push new_min to _min_stack

        # 1. pop _stack
        # 2. pop _min_stack

        pass
```

### 13.2 The code

```python
class MinStack:
    def __init__(self) -> None:
        self._stack: list[int] = []
        self._min_stack: list[int] = []         # parallel min tracker

    def push(self, val: int) -> None:
        self._stack.append(val)
        # The new running min is min(val, previous min)
        new_min = val if not self._min_stack else min(val, self._min_stack[-1])
        self._min_stack.append(new_min)

    def pop(self) -> None:
        self._stack.pop()
        self._min_stack.pop()

    def top(self) -> int:
        return self._stack[-1]

    def getMin(self) -> int:
        return self._min_stack[-1]
```

### 13.3 What to say while typing

- *"Two parallel lists — `_stack` for values, `_min_stack` for running mins."*
- *"On push — append val. Compute new_min as val if min_stack is empty, else min of val and previous min. Append new_min."*
- *"On pop — pop both stacks together to keep them aligned."*
- *"top — last element of `_stack`."*
- *"getMin — last element of `_min_stack`."*

### 13.4 Naming choices

- `_stack`, `_min_stack` — leading underscore = "private internal state"
- Keeping them as `list` (Python) instead of writing a real Stack class — `list.append` and `list.pop` are already O(1) at the end
- Method names match LeetCode signature exactly (`push`, `pop`, `top`, `getMin`)

> 💡 **Subtle point worth mentioning to interviewer:**
> *"In Python, list append and pop from the end are O of one amortized, so a list is effectively a stack."*

## 🟦 TEST (3 min)

### 15. Dry run

> **You:** *"With push(-2), push(0), push(-3), getMin, pop, top, getMin:*
>
> *push(-2): stack=[-2], min_stack=[-2]*
> *push(0):  stack=[-2,0], min_stack=[-2,-2] (min stays -2)*
> *push(-3): stack=[-2,0,-3], min_stack=[-2,-2,-3] (new min -3)*
> *getMin: min_stack[-1] = -3. ✓*
> *pop: stack=[-2,0], min_stack=[-2,-2]*
> *top: stack[-1] = 0. ✓*
> *getMin: min_stack[-1] = -2. ✓"*

### 16. Edge cases

> **You:** *"Edge cases:*
> *- First push on empty stack — min_stack is empty, so new_min = val. Works.*
> *- All equal values like push(3), push(3), push(3) — min_stack = [3, 3, 3]. Pops correctly down. Works.*
> *- Strictly decreasing pushes — min_stack tracks each new min. Works.*
> *- Strictly increasing pushes — min_stack repeats the same first value. Works.*
> *- Single element + pop + getMin — problem says ops are valid, so we won't hit empty-stack case."*

### 17. Off-by-one

> **You:** *"Both stacks always have the same length, so they stay aligned. No index arithmetic, no off-by-one risk."*

## 🟦 COMPLEXITY (1 min)

### 18. Final

> **You:** *"All four operations are O of one — list append, pop, and indexing are constant time.*
> *Space is O of n, where n is the number of pushed elements. The min_stack adds one entry per push.*
>
> *Trade-off: I could optimize space by only pushing to min_stack when val ≤ current_min,
> and on pop, only pop min_stack if the popped value equals the current min.
> That saves space when many duplicate-large values are pushed,
> but the comparison adds slight complexity. The current approach is simpler and equally O of one."*

## 📝 Final Code (copy-paste ready)

### Recommended (auxiliary stack)

```python
class MinStack:
    def __init__(self) -> None:
        self._stack: list[int] = []
        self._min_stack: list[int] = []

    def push(self, val: int) -> None:
        self._stack.append(val)
        new_min = val if not self._min_stack else min(val, self._min_stack[-1])
        self._min_stack.append(new_min)

    def pop(self) -> None:
        self._stack.pop()
        self._min_stack.pop()

    def top(self) -> int:
        return self._stack[-1]

    def getMin(self) -> int:
        return self._min_stack[-1]
```

**Complexity:**

- All operations: **O(1)**
- Space: **O(n)**

### Alternative — single stack of tuples

Mention only if asked *"can you do it with a single stack?"*

```python
class MinStack:
    def __init__(self) -> None:
        # Each entry: (value, current_min_at_this_point)
        self._stack: list[tuple[int, int]] = []

    def push(self, val: int) -> None:
        new_min = val if not self._stack else min(val, self._stack[-1][1])
        self._stack.append((val, new_min))

    def pop(self) -> None:
        self._stack.pop()

    def top(self) -> int:
        return self._stack[-1][0]

    def getMin(self) -> int:
        return self._stack[-1][1]
```

> 💡 **Same complexity.** The two-stack version is more intuitive to explain; the tuple version is slightly less memory due to one fewer list overhead. Both are accepted.

## 🧠 Pattern: Auxiliary Stack — what to remember

| When to use | Need O(1) access to a "running statistic" (min, max, sum) on top of stack semantics |
|---|---|
| Why parallel stack works | Stores the answer for each "level" of the main stack — pop reverts naturally |
| Trade-off | Extra O(n) space for O(1) statistic queries |
| Same template applies to | Max Stack, Stack with both min and max, Stack with running median |
